# ModuShop

ModuShop — це навчальний проєкт у стилі **Modular Monolith (Modulith)**, реалізований на платформі **.NET 10** в межах практикуму **№2.4 — Модульний моноліт**.

Проєкт побудований на основі підходів, продемонстрованих у **NimblePros Modular Monolith Workshop**, та адаптований під домен **онлайн-магазину**. Система розгортається як єдиний застосунок, але логічно поділена на ізольовані модулі з чіткими межами відповідальності.

## Мета проєкту

Мета проєкту — практично реалізувати архітектуру **модульного моноліту** з використанням сучасного .NET-стеку, продемонструвавши:

- побудову одного застосунку з чітко розділеними модулями;
- ізоляцію відповідальностей між частинами системи;
- використання **contracts-проєктів** для підготовки до слабшої зв’язності;
- централізовану точку входу через **ASP.NET Core Minimal APIs**;
- локальну оркестрацію і спостережуваність через **.NET Aspire**.

## Предметна область

ModuShop моделює базовий сценарій **інтернет-магазину**. Користувач може:

- зареєструватися;
- увійти в систему;
- переглядати каталог товарів;
- створювати замовлення;
- отримувати підтвердження замовлення.

Адміністратор може:

- створювати категорії;
- додавати товари;
- переглядати звіти по продажах.

## Реалізовані модулі

### Users
Модуль відповідає за:
- реєстрацію користувачів;
- логін;
- отримання користувача по ID.

### Catalog
Модуль відповідає за:
- створення категорій;
- створення товарів;
- отримання списку товарів;
- перегляд товару по ID.

### Orders
Модуль відповідає за:
- створення замовлення;
- зберігання позицій замовлення;
- отримання замовлення по ID;
- отримання замовлень користувача.

### Emailing
Модуль відповідає за:
- надсилання підтвердження замовлення;
- у поточній реалізації використовується **fake/logging email sender**, який імітує надсилання листа через логування.

### Reporting
Модуль відповідає за:
- зведення продажів;
- формування списку топових товарів;
- групування замовлень по днях.

## Архітектурний підхід

Проєкт реалізовано як **modular monolith**:

- застосунок запускається як **єдиний процес**;
- бізнес-логіка розділена на **окремі модулі**;
- кожен модуль має власний проєкт реалізації;
- для кожного модуля підготовлено окремий `*.Contracts` проєкт;
- модулі підключаються через власні `ModuleExtensions`;
- інфраструктурна конфігурація збирається в точці входу застосунку.

Хоча система фізично розгортається як один застосунок, її структура орієнтована на:
- модульність;
- локалізацію змін;
- простіше тестування;
- можливість подальшої еволюції окремих модулів у самостійні сервіси.

## Коротке пояснення архітектурних рішень

### 1. Modular Monolith замість мікросервісів
Для практикуму обрано **модульний моноліт**, оскільки він дозволяє:
- уникнути інфраструктурної складності мікросервісів;
- зберегти зрозумілу модель розгортання;
- одночасно продемонструвати bounded contexts і чіткі межі відповідальності.

### 2. ASP.NET Core Minimal APIs + FastEndpoints
Minimal APIs і FastEndpoints використані як проста та сучасна точка входу в систему. Це дозволяє будувати легкі API-endpoint’и без зайвої шаблонності.

### 3. .NET Aspire
**AppHost** і **ServiceDefaults** використовуються для локальної оркестрації, запуску застосунку та підключення SQL Server у єдиному середовищі розробки.

### 4. SQL Server + окремі схеми
Проєкт використовує одну SQL Server базу, але з логічним поділом на схеми:
- `users`
- `catalog`
- `orders`

Такий підхід добре відповідає ідеї modular monolith: БД одна, але межі модулів зберігаються.

### 5. Contracts-проєкти
Для кожного модуля створено окремий `Contracts`-проєкт. У поточній версії вони використовуються частково, але така структура закладає основу для подальшого послаблення зв’язності між модулями.

### 6. Окремий модуль Emailing
Функція email-повідомлень винесена в окремий модуль, щоб показати виділення окремої бізнес-відповідальності навіть у межах моноліту.

### 7. Окремий модуль Reporting
Аналітика винесена в окремий модуль читання, що дозволяє не змішувати transactional use cases із reporting-запитами.

## Інструкція запуску

### 1. Клонування репозиторію

```bash
git clone https://github.com/YOUR_USERNAME/practicum-2-4.git
cd practicum-2-4/src
```

### 2. Відновлення залежностей
```bash
dotnet restore
```
### 3. Переконайтесь, що Docker Desktop запущений

Перед стартом застосунку потрібно, щоб Docker був активний, оскільки SQL Server піднімається через Aspire.

### 4. Запуск застосунку
```bash
dotnet run --project ModuShop.AppHost
```
### 5. Відкриття Swagger

Після запуску Aspire покаже адреси доступних сервісів.
Потрібно відкрити webapi і перейти до Swagger UI за адресою на кшталт:

https://localhost:xxxx/swagger

- Зареєструвати користувача через POST /register

- Перевірити звіти через GET /api/reports/sales-summary
## База даних

У системі використовується SQL Server.
Для збереження модульності дані розділено по схемах:
- users
- catalog
- orders

Це дозволяє уникнути повного змішування таблиць між модулями і краще підтримує bounded contexts у межах одного моноліту.

## Поточні обмеження

Це навчальний проєкт, тому деякі рішення свідомо спрощені:

- email sender є логічним/fake, а не реальним SMTP-провайдером;
- частина міжмодульної взаємодії виконується напряму, а не через повністю оформлені application services;
- contracts-проєкти підготовлені як архітектурна основа, але можуть бути розвинуті далі;
- звіти реалізовані у спрощеному вигляді без окремої read-model бази.
